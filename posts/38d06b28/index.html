小米账号 Web 登录分析

发表于2026-05-17|更新于2026-06-02|ReverseProgramming

最近在做小米系服务自动化时，发现很多业务最终都绕不开小米账号体系。

无论后面要访问的是钱包、IoT、云服务，还是其它小米系业务，很多场景的第一步都是先拿到一个可用的小米账号 passToken。拿到 passToken 后，再根据目标业务的 sid 去换对应业务的 serviceToken。

这篇文章记录我分析小米账号 Web 登录的过程：从 serviceLogin 获取登录上下文，到 serviceLoginAuth2 提交账号密码，再到短信验证后的 serviceLoginAuth2/end 下发 passToken。

重点不是某个具体业务接口，而是“小米账号 Web 登录如何拿到 passToken”这条通用链路。

背景

在小米账号体系里，passToken 可以理解为账号层面的重要登录凭证。

它不是某个具体业务的 Cookie。

更准确地说：

也就是说，passToken 是账号层凭证，而 serviceToken 是业务层凭证。

不同业务有不同的 sid，例如：

同一个小米账号拿到 passToken 后，可以继续针对不同业务的 sid 走 STS 换票。

所以，把 Web 登录获取 passToken 这部分单独拆出来，是很有价值的。它可以被多个小米系项目复用。

  3. 把 AES key 通过 RSA 加密后放进请求头 EUI。

serviceLoginAuth2 的返回分支

serviceLoginAuth2 不是只有一种成功结果。

实际抓包中至少要处理三类分支。

返回情况含义后续处理
code=0 且有 passToken直接登录成功保存 userId/passToken
code=87001需要人机验证码进入验证码分支

这也是实现 Web 登录时最容易踩坑的地方。

浏览器指纹与设备态

Web 登录是否触发验证码，和设备态关系很大。

在自动化里，最重要的不是“每次随机”，而是“每个账号固定一套身份”。

为什么不要频繁随机

自动化里很常见的误区是：

每次运行都随机 UA、随机 deviceId、随机 fingerprint。

这并不一定更安全。

从风控视角看，一个账号如果每次登录都换设备、换浏览器、换环境，反而更像异常行为。

踩坑记录

1. 不能直接 POST serviceLoginAuth2

如果没有先请求 serviceLogin 拿到上下文，直接 POST 很容易失败。

尤其是 _sign、qs、callback 这些参数，都依赖服务端下发。

2. _sign 不要本地算

_sign 不是前端算法随便能算出来的字段。

它来自小米服务端下发的登录 URL。

自动化脚本应该解析它，而不是伪造它。

3. user 字段不是简单 Base64

user 是 AES-CBC 加密后的 Base64。

而 AES key 又通过 RSA 放进 EUI。

如果只做 Base64 或固定 AES key，都会失败。

4. code=0 不一定代表已经拿到 passToken

serviceLoginAuth2 返回 code=0 时，如果同时有 securityStatus=16，说明还需要身份验证。

此时没有 passToken。

必须继续走短信验证。

5. 短信验证成功也不是终点

verifyPhone 成功后返回的是 identity/result/check。

真正下发 passToken 的是最后的：

6. 验证码不一定代表加密错

code=87001 更多是风控分支。

如果带上正确验证码状态后同样的 user/EUI/hash 能成功，就说明加密逻辑没问题。

7. 同一账号不要频繁换设备态

设备态应该长期稳定。

尤其是：

同一个账号频繁变化这些字段，可能更容易触发短信或验证码。

总结

小米账号 Web 登录不是一个简单的账号密码 POST。

它至少包含四层东西：

  1. 服务端下发的登录上下文。
  2. 前端对 user 字段的加密和 EUI 请求头。
  3. 风控分支，包括人机验证码和手机验证。
  4. 最终 passToken 的落点。

当这条链路拆清楚后，passToken 获取逻辑就可以封装成独立模块。

这比每次随机参数更重要。

稳定的账号态、清晰的分支处理、可复用的登录模块，才是这类自动化项目能长期运行的基础。
