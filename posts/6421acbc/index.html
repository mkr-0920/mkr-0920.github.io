Android Root 模式下实现进程内存读写与硬件断点

发表于2026-04-20|更新于2026-06-02|Reverse

本文讨论的是在已授权 Root 环境下，如何设计一套用户态的 Android 进程调试与内存修改后端。内容面向安全研究、逆向分析、调试工具开发和自有设备上的工程验证。

1. 问题定义

在 Android 上做进程内存分析，一般需要实现这些能力：

  1. 枚举系统进程。
  2. 获取目标进程名称、cmdline、内存映射。
  3. 打开目标进程并建立逻辑句柄。
  4. 读取目标进程内存。
  5. 写入目标进程内存。
  6. 批量写入，用于数值冻结。
  7. 解析 /proc/<pid>/maps，过滤可读、可写、可执行区域。
  8. 搜索数值、字符串、AOB 特征码。
  9. 反汇编目标地址附近代码。
  10. 汇编 ARM64 指令并写回目标进程。
  11. 在目标进程内分配新内存，用于 code cave 或 trampoline。
  12. 安装硬件执行断点和访问断点。
  13. 读取硬件断点命中时的寄存器。
  14. 命中后让目标线程继续执行。

如果有内核驱动，这些能力可以交给内核模块完成。如果有内核级 syscall hook，也可以走内核扩展通道。但在纯 Root 模式下，通常只能依赖 Android/Linux 已有接口：

  • su
  • /proc
  • /proc/<pid>/mem
  • Unix domain socket
  • shared memory
  • ptrace
  • PTRACE_GETREGSET
  • PTRACE_SETREGSET
  • ARM64 hardware debug registers

所以 Root 模式的本质是：

用一个 root 权限的用户态守护进程，代替内核驱动完成敏感操作；普通 App 只负责 UI、业务逻辑和 IPC 调用。

2. 架构与启动流程

2.1 总体架构

推荐把系统拆成三层：

  1. App/UI 层

    • 负责悬浮窗、搜索界面、结果列表、冻结列表、断点界面。
    • 不直接访问 /proc/<pid>/mem。
    • 不直接 ptrace 目标进程。
    • 通过 JNI 或本地库调用统一接口。

  2. 本地桥接层

    • 对上提供稳定 API，例如 openProcess、readMem、writeMem、getMaps、addHardwareBreakpoint。
    • 对下封装 IPC 协议。
    • 负责共享内存映射、socket 连接、请求序列化、响应解析。

  3. Root Server 层

    • 通过 su 启动。
    • 拥有 root 权限。
    • 访问 /proc/<pid>/mem、/proc/<pid>/maps、/proc/<pid>/cmdline。
    • 使用 ptrace 操作目标进程线程。
    • 负责硬件断点安装、命中捕获、寄存器读取和线程恢复。

  • Root Server 是 root 权限进程。
  • 所有敏感操作集中在 Root Server。

2.3 为什么让 App 监听，Root Server 连接

常规 C/S 模型通常是 server 监听、client 连接。但 Root 模式下可以反过来：

  1. App 先创建 abstract Unix socket 并 bind/listen。
  2. App 通过 su 启动 root server。
  3. root server 用传入的 socket name 主动 connect App。

这样做有几个优点：

  1. socket name 由 App 生成，带当前 App PID，避免多实例冲突。
  2. App 可以设置连接超时，初始化失败能及时返回。
  3. root server 不需要在文件系统上创建 pathname socket。
  4. abstract socket 生命周期跟随 fd，不会留下文件残留。

4.8 数值搜索

Root 模式下搜索可以完全在 App 侧做，也可以下放到 Root Server。

4.8.1 App 侧搜索

流程：

  1. 获取 maps。
  2. 过滤可读区域。
  3. 分块调用 read_mem。
  4. 在 App 侧扫描 buffer。

优点：

  1. 实现简单。
  2. 搜索逻辑容易写 Kotlin/C++。
  3. Root Server 保持轻量。

缺点：

  1. 大量内存数据要从 server 传回 App。
  2. 首次搜索大范围内存时开销较大。

4.8.2 Server 侧搜索

流程：

  1. App 把搜索条件发给 server。
  2. server 直接读取 /proc/pid/mem 并扫描。
  3. server 只返回地址列表。

优点：

  1. 减少数据传输。
  2. 大范围搜索更快。

缺点：

  1. IPC 协议复杂。
  2. server 要实现各种数据类型、模糊搜索、范围过滤。
  3. 搜索结果可能很大，仍然需要设计传输方案。

工程上可以先做 App 侧搜索，性能不够再把核心扫描下放到 Root Server。

4.9 指针链扫描

指针链扫描一般依赖 maps 和内存读：

  1. 获取所有可读、可写区域。
  2. 按 pointer size 扫描可能的指针值。
  3. 建立 value -> address list 或反向索引。
  4. 从目标地址反向查找父指针。
  5. 限制最大层数和 offset 范围。
  6. 找到落在模块内的 root pointer。

Root 模式下需要重点处理性能：

  1. 分块读取 maps 区域。
  2. 避免每个指针一次 read_mem。
  3. 对大区域建立本地缓存。
  4. 控制最大结果数，防止 OOM。

指针扫描不是 Root Server 必须内置的功能，它更适合放在 App/native bridge 侧，复用基础读内存能力即可。

4.11 远程内存分配

远程内存分配是 Root 模式下较难的一项能力。目标是让目标进程执行：

常见方案有三种。

4.11.1 ptrace 注入 syscall

基本流程：

  1. 选择目标线程。
  2. PTRACE_ATTACH。
  3. waitpid 等线程停止。
  4. PTRACE_GETREGSET(NT_PRSTATUS) 保存寄存器。
  5. 设置 ARM64 syscall 参数：
    • x0 = 0
    • x1 = size
    • x2 = PROT_READ | PROT_WRITE | PROT_EXEC
    • x3 = MAP_PRIVATE | MAP_ANONYMOUS
    • x4 = -1
    • x5 = 0
    • x8 = __NR_mmap
  6. 让线程执行一条 svc #0。
  7. 读取 x0 作为返回地址。
  8. 恢复寄存器和原始代码。
  9. detach/continue。

难点：

  1. 需要找到可执行位置写入 svc #0。
  2. 需要保存和恢复原始指令。
  3. 要处理线程原始信号状态。
  4. 如果目标正在执行敏感代码，强行改寄存器可能有风险。

4.11.2 调用目标进程已有 libc mmap

流程：

  1. 解析目标进程 maps，找到 libc base。
  2. 根据本机 libc 中 mmap 偏移计算目标进程 mmap 地址。
  3. 构造远程调用。

难点：

  1. ASLR。
  2. 不同 Android 版本符号差异。
  3. PAC/BTI 等机制。
  4. 调用约定和栈状态。

4.12 ARM64 Hook 的基本流程

完整 hook 通常需要：

  1. 读取目标地址附近原始指令。
  2. 反汇编，计算至少覆盖多少字节才能放下跳转桩。
  3. 远程分配 code cave。
  4. 把用户 patch 代码写入 code cave。
  5. 把被覆盖的原始指令重定位到 code cave。
  6. 在 code cave 尾部跳回 target + overwrite_len。
  7. 在原始地址写入跳转到 code cave 的入口桩。
  8. 验证写入结果。

6. 工程化与稳定性

6.1 错误码设计

Root Server 不建议所有失败都返回 1。最好返回具体 errno 或自定义负数。

推荐：

错误含义
EPERMRoot 权限不足，或 ptrace attach 被拒绝
ENOENTPID 不存在，或 /proc 路径不存在
EACCESmaps/mem 权限不足
EINVAL参数非法
EBUSY硬件断点 slot 已满
EIOregset 写入失败或内存读写失败
ETIMEDOUT暂无断点命中

6.2 SELinux 与 Android 版本差异

Root 模式并不意味着所有接口都稳定可用。常见影响因素：

  1. SELinux enforcing/permissive。
  2. su 实现差异。
  3. Android 版本对 /proc/<pid>/mem 的限制。
  4. 目标进程 dumpable 状态。
  5. 目标是否被其他 tracer attach。
  6. 内核是否允许对应 ptrace regset。
  7. SoC 暴露的 hardware breakpoint/watchpoint 数量。

启动时可以做自检：

  1. su -c id 是否成功。
  2. Root Server 是否能连接 socket。
  3. REQ_PING 是否成功。
  4. 能否读取自身或测试进程 /proc/pid/maps。
  5. 能否读取一段已知可读内存。
  6. 能否 attach 测试进程并查询 NT_ARM_HW_WATCH。

6.3 安全与生命周期

Root Server 拥有很高权限，必须控制生命周期：

  1. socket name 使用随机后缀或当前 PID。
  2. App 退出时主动关闭 socket。
  3. server 收到断开后清理 ptrace 状态。
  4. 不暴露固定公开 socket name。
  5. 不接受未知客户端连接。
  6. 协议里可以加入一次性 token。

7. 总结

Android Root 模式下实现进程内存工具，核心不是一条 su 命令，而是一套完整的用户态后端架构：

  1. 用 su 启动 root server。
  2. 用 abstract Unix socket 做控制通道。
  3. 用 SCM_RIGHTS 传 fd。
  4. 用 shared memory 承载大块读写数据。
  5. 用 /proc/<pid>/mem 做跨进程内存读写。
  6. 用 /proc/<pid>/maps 做内存区域枚举。
  7. 用 /proc/<pid>/task 获取线程。
  8. 用 ptrace 操作 ARM64 hardware breakpoint/watchpoint。
  9. 用 PTRACE_GETREGSET 采集命中寄存器。
  10. 用临时禁用断点 + 单步 + 恢复断点解决重复命中问题。

这种方案不需要内核驱动，部署成本低，适合 Root 设备上的调试和研究。但它也有天然限制：

  1. 依赖 su 和 SELinux 环境。
  2. /proc/pid/mem 在不同系统上行为可能不同。
  3. ptrace 会影响目标线程状态，也可能被检测。
  4. 硬件断点必须处理多线程、clone、命中恢复。
  5. 远程分配内存和 trampoline hook 实现复杂。

工程上最合理的做法是：先把进程枚举、maps、读写、批量写这条主链路做稳定，再逐步增加硬件断点、远程 mmap 和 inline hook。Root 模式可以做到功能很完整，但每一项能力都应该明确边界、错误码和回滚路径。
