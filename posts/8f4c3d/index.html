x64dbg / x32dbg 使用教程：从界面到断点、跟踪与常用快捷键

发表于2026-05-29|更新于2026-06-07|Reverse

前言

x64dbg / x32dbg 是 Windows 平台上非常常用的动态调试器。它们适合用来观察程序运行时的状态，包括当前执行到哪条指令、寄存器里有什么、栈上保存了什么参数、某块内存什么时候被读取或修改，以及某个 API 是从哪里被调用的。

两者的关系很简单：

  • x64dbg：调试 64 位程序。
  • x32dbg：调试 32 位程序。
  • 两者界面、菜单、快捷键和工作流基本一致，只是寄存器名称、调用约定和地址宽度不同。

本文重点放在“怎么用调试器”上，不把篇幅放在汇编基础、PE 结构或反调试原理。阅读目标是：你能够打开一个程序，找到关键位置，下断点，单步分析，查看寄存器、栈、内存、字符串引用，并把分析过程保存下来。

只建议在自己编写的程序、开源程序、CTF/练习样本或已获得授权的目标上练习调试。调试器本身是学习和分析工具，具体使用场景需要遵守软件授权和法律要求。

x64dbg 与 x32dbg 怎么选

调试器位数必须和目标程序位数匹配：

目标程序使用调试器
32 位 EXE / DLLx32dbg
64 位 EXE / DLLx64dbg

如果不确定程序位数，可以用下面几种方式判断：

  1. 看任务管理器。32 位程序通常会显示为 xxx.exe *32。
  2. 用 PE 工具查看 Machine 字段。
  3. 直接尝试打开。位数不匹配时，x64dbg / x32dbg 会提示无法调试。
  4. 看程序安装目录。Program Files (x86) 中的程序通常是 32 位，但这不是绝对判断。

实际分析中经常会同时保留两个版本，因为很多老程序、游戏辅助工具、安装器、外壳程序仍然是 32 位。

安装与目录结构

x64dbg 通常不需要复杂安装，下载后解压即可使用。目录里常见内容如下：

文件或目录作用
x64dbg.exe启动 64 位调试器
x32dbg.exe启动 32 位调试器
release\x6464 位核心文件、插件、数据库
release\x3232 位核心文件、插件、数据库
plugins插件目录
scripts脚本目录
db调试数据库目录

建议做几个基础设置：

  • 字体调大一点，长时间看反汇编更舒服。
  • 使用固定宽度字体，例如 Consolas、JetBrains Mono、Cascadia Mono。
  • 开启或保留自动保存数据库，方便下次继续分析。
  • 熟悉右键菜单。x64dbg 的很多功能入口都在反汇编窗口、Dump 窗口和栈窗口的右键菜单里。

主界面总览

打开程序后，最常用的是 CPU 窗口。它一般由几个区域组成：

区域主要作用
Disassembly / 反汇编窗口查看当前代码指令
Registers / 寄存器窗口查看和修改寄存器
Dump / 内存窗口查看和修改内存
Stack / 栈窗口查看调用栈数据、返回地址、参数
Command / 命令栏输入调试命令
Log / 日志窗口查看调试器输出和命令结果

刚开始学习时，不需要把所有窗口都掌握。先记住一句话：

反汇编窗口看代码，寄存器窗口看状态，Dump 窗口看内存，Stack 窗口看调用现场。

Stack 栈窗口

栈窗口显示 RSP / ESP 附近的内存。它的作用包括：

  • 查看返回地址。
  • 查看函数参数。
  • 查看局部变量。
  • 查看保存的寄存器。
  • 判断当前函数是从哪里调用过来的。

x86 和 x64 的调用约定不一样：

架构常见传参方式
x86参数通常从右到左压入栈
x64前 4 个整数或指针参数通常放在 RCX、RDX、R8、R9，其余参数放在栈上

加载程序后的基本流程

打开目标程序后，x64dbg 通常不会直接停在你想看的业务代码，而是停在系统相关位置，例如 ntdll 内部的系统断点。

常见流程如下：

  1. 启动 x64dbg 或 x32dbg。
  2. 打开目标 EXE。
  3. 程序停在系统断点。
  4. 使用 Ctrl + F9 或菜单运行到返回，或使用 Alt + F9 运行到用户代码。
  5. 找到程序入口点、字符串、API 或感兴趣的模块。
  6. 设置断点。
  7. F9 运行，等断点命中。
  8. 使用 F7 / F8 单步分析。
  9. 查看寄存器、栈、Dump。
  10. 添加注释和标签。
  11. 保存数据库。

几个概念要先分清：

概念说明
系统断点调试器附加或启动程序后，系统模块中的初始暂停点
入口点PE 头中记录的程序入口，未必是源码里的 main
TLS 回调程序入口点之前可能执行的初始化回调
用户代码目标程序自身模块中的代码
系统代码ntdll.dll、kernel32.dll、KernelBase.dll 等系统模块中的代码

如果你发现自己一直在 ntdll、kernel32 里单步，大概率是进入系统 API 了。这时可以用 Ctrl + F9 执行到返回，或者用 Alt + F9 回到用户代码。

运行控制快捷键

运行控制是最基础也最重要的部分。

快捷键名称说明
F7Step Into步入。遇到 call 会进入被调用函数
F8Step Over步过。遇到 call 不进入函数，只执行完调用
F9Run继续运行，直到断点、异常或程序结束
F12Pause暂停正在运行的程序
Ctrl + F2Restart重启调试
Ctrl + F9Run until return执行到当前函数返回
Alt + F9Run until user code执行到用户代码

F7 和 F8 的区别

遇到普通指令时，F7 和 F8 看起来差别不大。遇到 call 时区别很明显：

  • 按 F7：进入 140001000 这个函数内部。
  • 按 F8：不进入函数内部，直接执行完整个 call，停在下一条指令。

什么时候用 F7：

  • 你怀疑这个函数就是关键逻辑。
  • 你想看函数内部怎么处理参数。
  • 你想观察返回值是怎么产生的。

什么时候用 F8：

  • 这个函数是系统 API，不想进去。
  • 这个函数不是当前分析重点。
  • 你只关心调用后的返回值。

Ctrl + F9 的用途

Ctrl + F9 会执行到当前函数返回，适合在这些场景使用：

  • 不小心 F7 进入了系统 API。
  • 已经看完当前函数，想快速回到调用者。
  • 进入了很长的循环或初始化函数，想跳出去。

执行到返回后，通常还需要再按一次 F8，让程序真正越过 ret 回到调用点后面。

断点功能详解

断点是动态调试的核心。简单说，断点就是让程序运行到某个位置时自动暂停。

x64dbg 常用断点类型：

类型适合场景
软件断点在某条指令地址暂停
硬件断点监控执行、读、写，数量有限
内存断点监控某个内存页的访问
条件断点满足条件才暂停
日志断点命中时记录信息，不一定暂停

软件断点

软件断点最常用，快捷键是 F2。

操作：

  1. 在反汇编窗口选中一条指令。
  2. 按 F2。
  3. 地址左侧出现红点，表示断点已设置。
  4. 再按一次 F2 可以取消。

软件断点适合：

  • 函数入口。
  • 关键分支。
  • API 调用前后。
  • 字符串引用位置。
  • 可疑逻辑附近。

软件断点的基本原理是把目标位置的机器码临时替换为断点指令。调试器命中后再恢复原指令并暂停。平时使用时不需要关心这些细节，但要知道：某些程序会检查代码段是否被修改，这时软件断点可能被发现。

硬件断点

硬件断点使用 CPU 调试寄存器实现，不直接修改目标代码。它适合监控“谁执行了这里”或“谁访问了这块内存”。

常见类型：

类型作用
Execute执行到该地址时暂停
Read读取该地址时暂停
Write写入该地址时暂停
Access读取或写入该地址时暂停

入口通常是右键菜单：

硬件断点适合：

  • 代码被校验，不方便下软件断点。
  • 想知道某个变量是谁修改的。
  • 想观察某个对象字段什么时候被访问。
  • 想定位回调或虚函数调用。

注意事项：

  • 硬件断点数量有限，通常只有 4 个调试寄存器可用。
  • 对内存读写设置硬件断点时，要注意监控长度。
  • 如果地址经常变化，重启程序后可能需要重新定位。

内存断点

内存断点以页面为单位工作，适合追踪一整块内存的访问。常见用途是定位：

  • 谁读取了这个字符串。
  • 谁修改了这个缓冲区。
  • 谁访问了这个结构体。
  • 某块解密后的数据从哪里被使用。

操作思路：

  1. 在 Dump 窗口定位到目标内存。
  2. 右键设置 Memory Breakpoint。
  3. 选择访问类型，例如 Read、Write、Access。
  4. 运行程序。
  5. 命中后查看当前指令和调用栈。

内存断点比硬件断点覆盖范围更大，但也更容易频繁命中。使用时最好先缩小目标范围，比如先通过字符串、对象指针或缓冲区地址定位到更明确的位置。

断点管理

断点多了以后，需要用断点窗口管理。通常可以从菜单打开：

断点窗口里可以：

  • 启用断点。
  • 禁用断点。
  • 删除断点。
  • 查看断点类型。

  • 修改命中后的日志或命令。

建议养成习惯：

  • 暂时不用但以后可能还要用的断点设置为禁用。

  • 不要在大量循环内部随手下普通断点，否则很容易频繁暂停。

搜索功能

搜索是从“完全不知道代码在哪”到“找到可疑位置”的关键路径。x64dbg 里常用搜索包括字符串、常量、字节特征、引用和 API。

搜索字符串

字符串是最适合入门的定位方式。很多程序会包含错误提示、按钮文本、日志文本、路径、URL、配置字段等字符串。

常见流程：

  1. 在 CPU 窗口或模块上右键。
  2. 选择搜索字符串。
  3. 在字符串列表中搜索关键文本。
  4. 双击字符串跳转。
  5. 查看对该字符串的引用。
  6. 在引用位置下断点。
  7. 运行程序，触发对应功能。

字符串分析思路：

  • 错误提示字符串附近通常有判断逻辑。
  • 菜单文本、按钮文本附近可能有 UI 初始化逻辑。
  • URL、接口路径附近可能有网络请求逻辑。
  • 日志字符串附近通常能看到上下文信息。

如果搜不到字符串，可能有几个原因：

  • 字符串是运行时动态生成的。
  • 字符串被加密或压缩。
  • 使用了 Unicode，但你只搜了 ASCII。
  • 程序资源在资源段，不在普通代码引用里。
  • 目标代码还没被解包或加载。

搜索 API

如果知道目标行为，可以从 API 反推。

  5. 查看调用参数和返回值。

References 引用窗口

References 窗口用于展示搜索结果、字符串引用、函数引用、跳转引用等。它是分析路径的中转站。

常见用途：

  • 找到字符串后查看“哪里用了它”。
  • 找到 API 后查看“哪里调用了它”。
  • 找到函数后查看“谁调用了这个函数”。
  • 找到地址后查看“哪些地方跳转到这里”。

使用方法：

  1. 在某个地址、字符串、函数或 API 上右键。
  2. 选择查找引用。
  3. References 窗口出现结果。
  4. 双击结果跳转。
  5. 在可疑位置下断点。

Memory Map 内存布局

Memory Map 窗口显示当前进程的内存区域，包括模块、基址、大小、权限等。

常见字段：

字段说明
Base内存区域基址
Size区域大小
Protect内存权限
Type内存类型
Module所属模块
Info附加信息

常见权限：

权限说明
R可读
W可写
X可执行
ER可执行、可读
ERW可执行、可读、可写

分析时重点关注：

  • 当前地址属于哪个模块。
  • 目标程序主模块的基址是多少。
  • 哪些区域是可执行的。
  • 是否存在可疑的 ERW 内存。
  • DLL 是否已经被加载。

如果你在反汇编窗口看到一个地址，不确定它属于谁，可以在 Memory Map 中查找该地址所属区域。这样能判断当前是在主程序、插件 DLL、系统 DLL，还是动态申请的内存中。

Call Stack 调用栈

Call Stack 窗口显示当前调用链。它回答的问题是：

程序是怎么走到这里的？

例如你在 WriteFile 断点命中，当前代码停在系统 API 内部。此时看调用栈，可以看到是谁调用了 WriteFile，再往上是谁调用了这个函数。

常见用途：

  • 从系统 API 回溯到用户代码。
  • 找到关键函数的调用来源。
  • 判断当前逻辑属于哪个功能流程。
  • 在崩溃或异常时定位调用路径。

使用建议：

  • 命中 API 断点后，第一时间看调用栈。
  • 双击用户模块中的返回地址，跳回自己的程序代码。
  • 如果调用栈不完整，可能是栈被破坏、函数省略帧指针，或代码做了特殊跳转。

  • 修改结构体字段。

  • 修改配置值。

Patch 补丁功能

运行时修改只影响内存。如果要把修改写回文件，需要使用 Patch 功能。

常见流程：

  1. 在反汇编窗口修改指令。
  2. 右键选择 Patch 相关功能。
  3. 打开补丁列表。
  4. 检查修改地址、原始字节、新字节。
  5. 选择保存到文件。

注意事项：

  • 保存补丁前一定要确认修改的是目标模块，不是系统 DLL。
  • 如果目标程序有校验、签名或完整性检查，直接改文件可能导致程序无法运行。
  • 学习时建议先备份原文件。
  • 不要在未授权软件上做破解、绕过授权或破坏性修改。

线程、句柄与异常相关窗口

除了 CPU 主窗口，还有一些辅助窗口在特定场景下很有用。

什么时候看线程窗口：

  • 程序有多个工作线程。
  • 断点命中在非主线程。

什么时候看异常窗口或 SEH：

  • 程序触发异常。
  • 程序用异常做控制流。
  • 单步时经常停在异常处理相关位置。

什么时候看句柄窗口：

  • 想观察文件、注册表、互斥体、事件、进程、线程句柄。
  • 分析程序是否打开了某个文件或设备。

常见问题

为什么打开程序后停在 ntdll？

这是正常现象。调试器启动程序后，通常先停在系统断点。此时还没到目标程序的业务逻辑。可以使用 Alt + F9 运行到用户代码，也可以通过字符串、API、入口点等方式定位。

为什么 F7 后进入了系统 DLL？

因为当前指令调用了系统 API 或运行库函数。使用 F7 会进入函数内部。如果不想进入，应该用 F8。如果已经进入了，可以用 Ctrl + F9 执行到返回。

为什么断点没有命中？

可能原因：

  • 断点下在了错误模块。
  • 目标代码没有执行到。
  • 程序位数和调试器不匹配。
  • 地址受 ASLR 影响，重启后变了。
  • 代码是动态生成或后来才加载的。
  • 断点被程序检测或清除了。

为什么找不到字符串？

可能原因：

  • 字符串是 Unicode，不是 ASCII。
  • 字符串在资源段里。
  • 字符串被加密。
  • 字符串运行时才生成。
  • 目标模块还没加载。

修改指令后为什么重启失效？

因为你只是修改了当前进程内存。程序重启后会从磁盘重新加载原始文件。要永久修改，需要使用 Patch 功能保存到文件。

x64dbg 可以调试 32 位程序吗？

一般不要这样做。32 位程序用 x32dbg，64 位程序用 x64dbg。调试器位数和目标程序位数保持一致最省事。

如何快速回到用户代码？

常用方法：

  • Alt + F9 运行到用户代码。
  • 在 Call Stack 中双击用户模块返回地址。
  • 在 Memory Map 中找到主模块地址范围，再判断当前代码是否属于主模块。

软件断点和硬件断点怎么选？

普通代码位置用软件断点。代码不能修改、担心被检测、需要监控读写时用硬件断点。硬件断点数量有限，不适合大量使用。
