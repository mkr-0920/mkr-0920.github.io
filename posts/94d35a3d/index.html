非荣耀手机做荣耀钱包任务

发表于2026-06-01|更新于2026-07-02|Reverseprogramming

之前分析荣耀钱包 VIP 活动时，Python 侧主要解决的是账号态、钱包短票、营销活动接口和广告任务链路。

但如果把荣耀钱包直接装到非荣耀手机上，会遇到另一个问题：账号能登录，接口也能抓到，但活动页和任务链路里有不少设备环境不是普通 Android 手机天然具备的。

这些缺失不一定会让 App 直接崩溃。更常见的表现是：

  • H5 活动页能打开，但设备信息字段为空或长度不对。
  • Cookie 里缺少钱包容器注入的设备名片。
  • 广告 SDK 能初始化，但签名材料和真实请求结构不清楚。
  • 应用下载、打开、留存任务判断当前环境不支持。
  • 跳转到应用市场、下载服务或外部页面时，任务状态不一定能推进。

hook.js 解决的不是账号登录，也不是直接发奖。

它的定位更像端侧环境层：在非荣耀手机上观察荣耀钱包客户端到底要什么设备态，并在必要位置补齐缺失字段，让后续 Python 侧复刻业务链路时有可靠依据。

整体思路：先让 App 把设备态说清楚

非荣耀手机做荣耀钱包任务，不能只看网络请求。

荣耀钱包内部至少有几套设备上下文：

层级作用
H5 容器给活动页注入 Cookie、deviceinfo、JSBridge 设备名片
原生网络层构造钱包 API 的基础设备信息
广告 SDK负责广告拉取、签名、曝光、播放和应用类任务
下载服务负责应用下载、安装、打开、留存相关能力判断
Activity 跳转负责市场页、详情页、外部 App 打开等行为

三类 Hook：日志、修补、强制能力

整个 hook.js 可以按行为分成三类。

第一类是纯日志：

这些 Hook 不改变行为，只负责把隐藏在 App 内部的状态打印出来。

第二类是字段修补：

这类 Hook 只补齐非荣耀机上缺失或异常的设备字段。

第三类是能力强制：

这类 Hook 会改变端侧判断结果，目的是让应用类任务链路继续往后走，方便观察后续状态。

和 Python 侧的关系

hook.js 不是最终任务执行器。

它主要给 Python 侧提供三类信息：

  1. H5 和原生层需要哪些设备字段。
  2. 广告 SDK 的签名原文和 x-token 生成方式。
  3. 应用任务依赖哪些端侧环境判断。

验证顺序

调试这类 Hook 时，比较稳的验证顺序是：

如果第一步没有，说明 Hook 没挂上。

如果 Cookie 和 JSBridge 都没有，说明活动页还没进入 H5 容器业务流程。

如果广告 SDK 签名没有出现，说明当前操作还没触发广告拉取。

如果应用留存相关日志没有出现，说明还没进入应用类任务路径。

按这个顺序排查，比在所有日志里乱找要快很多。

局限性

这套 Hook 有几个明显边界。

第一，它依赖荣耀钱包当前版本的类名和方法签名。

App 升级后，如果混淆名或方法签名变化，Hook 点就需要重新确认。

第二，它只能解决端侧环境问题。

服务端仍然会校验账号态、钱包短票、活动任务状态、广告行为、设备一致性和领取上限。

第三，能力强制不是万能的。

把“荣耀市场已安装”返回 true，只能让端侧继续走后续流程，不代表服务端一定认可任务完成。

第四，非荣耀手机终究不是荣耀系统环境。

有些能力可以通过 Hook 补齐，有些能力只能观测后在 Python 侧复刻，有些能力如果依赖系统级服务或硬件可信环境，就不能简单靠 Java 层 Hook 解决。

总结

非荣耀手机做荣耀钱包任务，难点不在“App 能不能装上”，而在活动链路需要一整套荣耀钱包环境。

hook.js 实际做了三件事：

它让非荣耀设备上的荣耀钱包尽量表现出活动页需要的设备上下文，同时把广告 SDK 和应用任务链路里的关键状态打印出来。
