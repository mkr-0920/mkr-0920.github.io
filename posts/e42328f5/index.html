Android 内核驱动模式下实现进程内存读写与硬件断点

发表于2026-04-30|更新于2026-06-02|ReverseLinux

本文讨论的是在已授权测试设备上，通过 Android/Linux 内核模块实现进程调试与内存修改后端的工程方案。内容面向内核调试、安全研究、逆向分析和自有设备上的工具开发。

1. 问题定义

Root 模式可以通过 /proc/<pid>/mem 和 ptrace 实现一部分进程调试能力，但它有几个天然限制：

  1. 依赖 su 和 SELinux 环境。
  2. /proc/<pid>/mem 在不同 Android 版本上行为不完全一致。
  3. 硬件断点依赖 ptrace，会影响目标线程状态。
  4. 远程分配内存、修改页权限、跨线程硬件断点同步都比较麻烦。

内核驱动模式的思路不同：

把敏感操作放进内核模块中，由用户态通过一个受控通信接口发命令，驱动在内核态完成进程枚举、VMA 遍历、跨进程内存访问、远程 mmap 和硬件断点管理。

一个完整的驱动模式后端通常需要实现这些能力：

  1. 创建用户态通信节点。
  2. 隐藏或弱化通信节点可见性。
  3. 定义命令协议。
  4. 建立目标进程句柄表。
  5. 读取目标进程内存。
  6. 写入目标进程内存。
  7. 批量写入，用于冻结。
  8. 枚举进程 PID。
  9. 枚举目标进程 VMA。
  10. 获取 cmdline。
  11. 检查虚拟地址是否映射物理页。
  12. 在目标进程 mm 中分配内存。
  13. 可选：修改目标进程 cred。
  14. 可选：隐藏内核模块。
  15. 安装硬件执行断点和访问断点。
  16. 捕获断点命中并返回寄存器。
  17. 让命中线程继续执行。

本文按这些能力拆解驱动模式的实现。

2. 架构与通信入口

2.1 总体架构

推荐把驱动模式拆成四层：

  1. App/UI 层

    • 负责悬浮窗、搜索界面、冻结列表、断点面板。
    • 调用统一 Native API。

  2. Native Bridge 层

    • 对上提供 openProcess、readMem、writeMem、getMaps、remoteAlloc、addHardwareBreakpoint 等接口。
    • 对下封装驱动通信协议。
    • 负责打包请求头和 payload。

  3. 内存读写驱动

    • 提供进程枚举、句柄管理、VMA 枚举、内存读写、远程 mmap。
    • 常用内核 API 包括 find_get_pid、get_pid_task、get_task_mm、mmap_read_lock、access_process_vm、vm_mmap。

  4. 硬件断点驱动

    • 管理目标进程硬件断点。
    • 推荐使用内核 perf_event 硬件断点框架。
    • 通过 overflow handler 捕获命中时的 pt_regs。

整体关系：

读写驱动和硬件断点驱动可以做在同一个模块里，也可以拆成两个模块。拆开的好处是职责清晰：

  • 内存读写模块只关心进程、VMA、读写和分配。
  • 硬件断点模块只关心 breakpoint/watchpoint、命中和恢复。

  1. 字符设备。

3.12 汇编 Patch 与 Trampoline

驱动模式拥有远程 mmap 后，就可以支持更完整的 ARM64 hook：

  1. 读取目标地址原始指令。
  2. 反汇编并计算覆盖长度。
  3. 调用 ALLOC_PROCESS_MEMORY 分配 code cave。
  4. 汇编用户 patch 代码。
  5. 重定位被覆盖原始指令。
  6. 在 code cave 尾部跳回原始函数。
  7. 在目标地址写入跳转到 code cave 的入口桩。
  8. 读回验证。

驱动模式更强，但成本也更高：

  1. 需要匹配目标内核。
  2. 可能需要关闭或适配 CFI。
  3. 模块加载可能受 Android 安全策略限制。
  4. 一旦内核代码出错，可能直接 panic。

5.6 稳定性建议

  1. 所有用户指针必须 copy_from_user/copy_to_user。
  2. 所有 handle 都做 magic/type 校验。
  3. 所有 task/mm/pid 引用都成对释放。
  4. VMA 遍历必须持有 mmap read lock。
  5. 每次内存读写最多按页处理。
  6. 命令失败返回负 errno。
  7. 模块卸载前释放所有 handle、perf event、workqueue。
  8. 硬件断点命中等待必须能被 close 唤醒。
  9. hide module、set root 这类高风险功能做成可选。
  10. debug printk 默认关闭，避免高频读写刷日志。

5.7 推荐实现顺序

从零实现驱动模式，建议按这个顺序：

  1. 编译最小内核模块。
  2. 创建 procfs 通信节点。
  3. 实现 PING。
  4. 实现请求头解析和错误码返回。
  5. 实现 OPEN_PROCESS/CLOSE_PROCESS 和 IDR handle 表。
  6. 实现 READ_PROCESS_MEMORY。
  7. 实现 WRITE_PROCESS_MEMORY。
  8. 实现 PID 列表。
  9. 实现 VMA count/list。
  10. 实现 cmdline 地址读取。
  11. 实现远程 mmap。
  12. 实现批量写。
  13. 接入用户态搜索/冻结/汇编 Patch。
  14. 单独实现 HWBP 模块。
  15. 实现 perf_event execute breakpoint。
  16. 实现 write/readwrite watchpoint。
  17. 实现命中寄存器返回。
  18. 实现 continue 和 rearm。
  19. 加入生命周期清理。
  20. 最后再考虑隐藏模块和权限提升。

不要一开始就写模块隐藏、cred 修改和硬件断点。先把基础通信和内存读写打稳，否则后续定位问题会非常困难。

6. 总结

Android 驱动模式的核心，是把用户态难以稳定完成的能力下沉到内核：

  1. procfs 节点提供轻量通信入口。
  2. read() 可以承载伪 ioctl 协议。
  3. IDR 句柄表避免暴露内核指针。
  4. find_get_pid/get_pid_task/get_task_mm 建立目标进程访问链。
  5. access_process_vm 完成跨进程虚拟内存读写。
  6. mm_struct 和 VMA 遍历替代 /proc/pid/maps 文本解析。
  7. use_mm/kthread_use_mm + vm_mmap 可以在目标进程地址空间分配内存。
  8. perf_event 硬件断点框架比手写 debug register 更适合工程实现。
  9. overflow handler 可以直接拿到 pt_regs，实现命中寄存器采集和修改。
  10. wait queue 可以把命中事件同步给用户态，再由用户态决定何时继续。

驱动模式的优势是能力完整、性能高、对 ptrace 依赖低；缺点是内核适配成本高，出错代价大。真正可用的实现，重点不是“能读写一次内存”，而是句柄生命周期、用户指针安全、内核版本适配、错误码、模块卸载清理和硬件断点重入控制都要处理完整。

如果只是做基础内存搜索，Root 模式已经够用；如果要实现稳定的远程分配、trampoline hook、低副作用硬件断点和高频冻结，驱动模式才是更完整的后端形态。
