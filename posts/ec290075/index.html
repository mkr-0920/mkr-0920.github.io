荣耀钱包 VIP 活动业务链路分析

发表于2026-06-04|更新于2026-07-02|Reverseprogramming

之前分析这类钱包活动时，最容易把问题想简单：抓到活动接口，请求能返回成功，就以为任务链路已经结束。

但荣耀钱包这边实际不是这样。

账号登录只是最外层。真正进入活动页之后，服务端看的不是单纯的账号密码登录态，而是 HnID 下发的长票、钱包 App 作用域票据、营销活动短票、H5 容器设备信息、广告 SDK 侧设备态、任务行为状态，以及最后的余额变化。

这篇文章记录荣耀钱包 VIP 活动业务侧的分析思路：如何从 HnID 登录态走到钱包业务态，如何处理 service_token 和 hshop_accesstoken，如何让 H5、原生网络层、广告 SDK 三套设备信息保持一致，如何拆开营销活动接口和广告行为链路，以及库存监控为什么可以复用同一套鉴权能力。

背景：HnID 登录态之后还有钱包业务态

荣耀账号体系里，账号密码登录拿到的凭证只是第一层。

荣耀钱包活动实际使用的是钱包业务域里的短期访问票据。这个票据不是直接由账号密码接口返回，而是需要通过 HnID 登录态继续换取。

整个链路可以概括成：

所以项目里最终形成了两个相对独立的状态：

  • 账号态：负责确认这个账号是谁，以及能不能继续换票。
  • 钱包业务态：负责确认这个账号在钱包活动域里能不能访问、执行任务、查询库存。

这两个状态不能混在一起。

service_token 可用，不代表活动接口一定可用；hshop_accesstoken 可用，也不代表广告或应用任务一定完成。后面的任务状态还要继续经过活动接口和广告行为链路校验。

钱包短票：hshop_accesstoken 才是活动接口入口

荣耀钱包营销活动接口主要跑在钱包营销业务域下。

活动请求里最关键的不是账号密码，也不是原始 service_token，而是换票之后得到的短期访问票据，也就是项目里统一放进 Header 的 hshop_accesstoken。

这里有一个比较重要的结论：

已有长票可用时，应该优先续杯短票，而不是每次都重新账号密码登录。

原因有三个：

  1. 账号密码登录可能触发短信二次验证。
  2. 重新登录会改变会话状态，长期运行时没有必要。
  3. 短票本身就有刷新链路，只要长票有效，直接换票更稳定。

因此实际入口是：

如果探活发现短票无效，才进入重新登录流程。

账号密码登录这一步更多是兜底能力，而不是每次运行的默认路径。

service_token：不能直接当钱包票用

项目里踩过的一个点是：HnID 下发的原始 service_token 和钱包 App 使用的票据并不是完全一回事。

账号体系里拿到的是母票。

进入钱包业务时，还需要按目标包名派生 App 作用域票据。

也就是说，请求短票时不能简单把原始长票塞进去。服务端会结合包名、client_id、设备 ID、授权头和票据上下文做校验。

如果账号登录成功但短票换取失败，问题通常在钱包作用域票据或 silent_token 参数。

如果短票换取成功但活动接口失败，问题通常在活动 Header、设备态或营销签名。

设备态：荣耀钱包里最容易被低估的部分

荣耀钱包这条链路里，设备态不是一个字段，而是一组互相关联的字段。

至少有三套地方会使用设备信息：

位置作用
H5 容器活动页 Cookie、deviceinfo、JSBridge 设备名片
原生网络层钱包 API 请求头里的设备信息
广告 SDK广告拉取、曝光、点击、播放、留存任务判断

这三套设备信息如果互相不一致，现象会非常随机：

  • 活动接口能打开，但任务状态不变。
  • 广告接口能返回素材，但领奖时不认可。
  • 某个账号第一次能跑，下一次提示新设备。
  • 多账号之间状态串扰，或者都表现成同一台设备。

所以当前项目的方向是：每个账号固定一套设备指纹。

常见设备字段包括：

字段说明
device_id主设备 ID，账号态和钱包态都会用到
oaid系统广告标识相关字段
honor_oaid荣耀广告 SDK 侧使用的标识
imsiSIM 相关设备字段
udidH5 容器和钱包设备名片里使用
dvid2钱包侧补充设备字段

这里的目标不是每次生成“看起来很随机”的设备，而是让同一个账号长期表现成同一台设备。

这也是多账号隔离的基础。

应用留存任务

应用留存任务比广告任务更复杂。

它不是简单看一次广告，而是模拟“下载 / 打开 / 使用 / 次日留存”这类行为。

因为次日任务依赖前一天留下的行为记录。不能每次运行都当成新任务，也不能只在内存里保存状态。

行为加密：不是所有上报都是明文 JSON

应用类任务里有一类行为上报 payload 不是普通 JSON 明文。

它会先构造行为对象，再经过加密后作为字段提交。

这类设计说明服务端不只是看“有没有请求”，还会检查行为数据结构本身。

所以分析这类接口时要特别注意两点：

  1. 明文对象字段是否完整。
  2. 加密结果是否和当前设备、任务、广告素材对应。

如果只复制一份旧 payload，短期可能碰巧成功，但换账号、换广告素材、换日期后很容易失效。

节奏控制

项目里有几类随机等待：

场景目的
广告观看等待满足任务要求的真实时长
任务轮次间隔避免连续快速请求
账号间隔降低批量执行的机械感
库存平台间隔模拟用户切换平台页签
库存轮询间隔降低长期监控频率

这里的重点不是“随机越多越好”，而是每个等待都要对应业务含义。

广告等待对应播放时长，账号间隔对应批量执行节奏，库存轮询对应监控频率。

无意义的 sleep 只会拖慢脚本，有业务含义的等待才值得保留。

通知模块有一个原则：失败不影响主流程。

领奖已经完成，通知失败最多只是提醒缺失，不能反过来让任务执行失败。

库存监控也是一样。告警发送失败可以打印日志，但不应该导致监控进程退出。

踩坑记录

1. service_token 和钱包短票不是同一个东西

最开始容易误以为账号登录拿到的长票可以直接访问钱包活动。

实际不行。

钱包活动接口依赖的是换票后的短期业务票据。长票只是刷新短票的材料。

2. 设备字段必须跨层一致

H5 Cookie、JSBridge 设备名片、原生网络层、广告 SDK 请求里的设备字段要尽量一致。

只改某一层会造成很难定位的问题：接口返回不一定报错，但任务状态就是不推进。

3. 营销签名和广告签名不能混用

活动网关和广告 SDK 是两套签名体系。

看起来都叫 timestamp、nonce、sign，但拼接方式、大小写、Header 集合和参与 body 都不同。

4. 接口成功不等于奖励到账

活动类接口经常出现请求成功但余额不变。

最终判断必须看余额差值。

这也是项目里任务循环能自动停止的关键。

5. 应用任务依赖历史行为

应用留存任务不是一次性请求。

它会依赖前一天行为、今天行为、广告素材、加密行为 payload 和服务端状态。不能把它当成普通广告任务处理。

总结

荣耀钱包这条链路真正复杂的地方，不是某一个接口，也不是某一个签名。

它复杂在多个状态同时存在：

项目最终要解决的，就是把这些状态拆清楚，然后在 Python 里按服务端认可的顺序重新组织起来。

单次请求成功只能说明某个点通了。

余额能对账、库存能监控、多账号能隔离、失败能归类，才说明整条业务链路真正闭环。
